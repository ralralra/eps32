# 7회차 — 최종 발표: 7주의 증명

> 다른 사람이 써도 되는 시스템일까요 — 테스트로 다듬고, 발표로 증명해요.
> **테스트 · 리허설 · 최종 발표 · 4교시 (50분 수업 + 10분 휴식)**

## 오늘의 목표

- [ ] 다른 팀의 시선으로 우리 시스템의 약점을 찾아 보완한다
- [ ] 문제 정의부터 확장 계획까지, 5단계 구조로 발표를 완성한다
- [ ] 동료의 프로젝트를 기준을 갖고 평가하고, 나의 7주를 성찰한다

## 시간표

| 교시 | 활동 |
|:---:|---|
| 1교시 | 교차 테스트 (옆 팀 시스템을 시나리오 카드로 사용) · 오류 보완 |
| 2교시 | 발표 슬라이드 완성 · 시연 리허설 2회 · 백업 영상 촬영 |
| 3교시 | 1~3팀 발표 (발표 5분 + 시연 4분 + 질의 2분) |
| 4교시 | 4~5팀 발표 · 상호평가 집계 · 성찰 · 시상 · 수료 |

---

## 1교시 — 교차 사용성 테스트

**만든 사람은 자기 시스템의 함정을 보지 못해요.**

- 만든 사람의 사용법: 정확한 순서로, 조심스럽게, 될 만한 것만 눌러요 — 버그가 숨을 곳이 많아요
- 처음 쓰는 사람의 사용법: 설명 없이 아무거나, 연달아, 이상한 순서로 — **진짜 사용 환경!**
- 그래서 오늘: 팀을 맞바꿔 옆 팀 시스템을 사용 — 실제 기업의 'QA·베타테스트' 과정

### 따라하기 ① 테스트 시나리오 카드 실행

→ 기록 양식: [`worksheets/테스트_시나리오_카드.md`](worksheets/테스트_시나리오_카드.md)

1. **정상 시나리오**: 안내 문구만 보고 기본 기능 사용 — 설명 없이 가능했나요?
2. **짓궂은 시나리오**: 버튼 연타 · 뒤로 가기 · 순서 바꿔 누르기 — 시스템이 버티나요?
3. **데이터 확인**: 방금 행동이 시트에 정확히 남았나요? 중복·누락은 없나요?
4. **관찰 기록**: 사용자가 머뭇거린 지점 = 개선 포인트. '어디서 3초 이상 멈췄나'를 적어요

> 오류 기록 양식: **[언제] [무엇을 했더니] [무엇이 일어났다] [기대한 것은]** — 이 4칸이 채워져야 고칠 수 있어요.

### 발견된 오류, 어디까지 고칠까 — 작게, 안전하게

- **고쳐요**: 오타 · 안내 문구 · 알림 타이밍 등 화면 수준의 작은 수리
- **고치지 않아요**: 구조를 건드리는 큰 수술 — 발표에서 '발견된 한계와 개선 계획'으로 말하는 게 더 멋있어요
- 판단 기준: 수리 후 핵심 시나리오 재테스트에 **10분 이상** 걸릴 것 같으면 건드리지 않아요

## 2교시 — 발표 준비와 리허설

**잘 만든 것과 잘 보여주는 것은 다른 기술이에요.**

### 발표 구성 — 5단계 스토리

기술 자랑이 아니라 **'문제 해결 이야기'**를 들려줘요 → 템플릿: [`worksheets/발표_구성_템플릿.md`](worksheets/발표_구성_템플릿.md)

1. **문제 정의 (1장)**: 우리 학교의 어떤 불편함에서 출발했나 — 사진 한 장이면 최고
2. **우리의 설계 (1~2장)**: 시스템 구조도 + 구조 B를 선택한 이유 (4회차 비교표가 근거!)
3. **시연 (실물 4분)**: 핵심 시나리오를 폰→장치→시트 순서로 — 발표의 심장
4. **우회와 배움 (1장)**: 'RFID 대신 웹 버튼을 쓴 이유' — **제약을 설계로 푼 이야기는 감점이 아니라 가점!**
5. **데이터와 확장 (1장)**: 쌓인 데이터·통계 보여주기 + '한 달 더 있다면?' 확장 계획

> ⏱ 시간 배분: 발표 5분 + 시연 4분 + 질의 2분. **리허설로 시간을 실측하지 않으면 반드시 초과해요.**

### 시연이 무대에서 배신하지 않게 하려면

1. **리허설 최소 2회**: 1회차=흐름 점검 / 2회차=시간 재기 + 역할(조작·해설) 확정
   - 그중 1회는 '처음 보는 사람'(다른 팀원) 앞에서 — 아는 사람 앞 리허설은 반쪽짜리!
2. **백업 영상 촬영 (전 팀 의무)**: 잘 되는 지금 모습을 폰으로 녹화 — **무대 위 WiFi는 언제나 배신할 수 있어요**
   - 폰과 노트북 양쪽에 저장 — 열어서 재생까지 테스트
3. **발표 환경 점검**: 핫스팟 연결 · 보드 전원 · 폰 화면 미러링(또는 카메라로 비추기) 리허설
4. **실패 대응 시나리오**: 시연이 멈추면 → 침착하게 "준비한 영상으로 보여드리겠습니다" — **이것도 실력!**
5. 부스 세팅(보드·폰 거치대·노트북 위치, 청중이 장치 반응을 볼 수 있는 각도)을 사진으로 남기기 — 본 발표 세팅이 3분 만에 끝나요

## 3~4교시 — 최종 발표

### 평가 기준 — 무엇을 보나요

결과물의 화려함보다 **'과정과 이해'**를 봐요:

| 영역 | 세부 기준 | 배점 |
|---|---|:---:|
| **과정** | 설계 문서의 구체성 · 진행 기록 · 역할 수행 · 문제 해결 과정 | 40 |
| **결과물** | 핵심 시나리오 완성도(엔드 투 엔드) · 데이터 기록과 통계 · 안정성 | 40 |
| **발표** | 5단계 구성 · 시연 성공(영상 대체 인정) · 질의 응답 · 시간 준수 | 20 |

> 우회 전략은 감점 요소가 아니에요 — **'제약 조건을 설계로 해결한 능력'**으로 오히려 평가에 반영돼요.

### 듣는 팀의 역할 — 상호평가

→ 평가지: [`worksheets/상호평가지.md`](worksheets/상호평가지.md)

1. **상호평가지 작성**: 발표마다 — 가장 인상적인 점 1 · 궁금한 점 1 · 별점(완성도/발표력)
2. **질문 규칙**: 팀당 최소 1개 질문 — "왜 그렇게 설계했나요?"류의 **과정 질문**이 좋은 질문
3. **집계와 시상**: 교사 평가 + 상호평가 합산 → 최우수 · 아이디어상 · 끈기상 · 데이터상 · 협업상
   - **모든 팀이 하나 이상의 상을 받아요 — 7주를 완주한 것 자체가 이미 수상 자격!**

### 성찰 — 세 가지 질문 (제출하는 마지막 산출물)

→ [`worksheets/성찰노트.md`](worksheets/성찰노트.md)

1. 가장 어려웠던 순간과 넘은 방법 — 에러 앞에서 무엇을 했나
2. 7주 전의 나에게 해줄 조언 — '설계 먼저 해' '한 번에 다 고치지 마' … 여러분의 조언이 다음 기수의 교재가 돼요
3. 더 배우고 싶어진 것 — 웹? 하드웨어? 데이터? AI?

### 이 다음은? — 확장의 갈림길

- **더 깊은 하드웨어**: 외부 센서·액추에이터 추가, 배터리 구동, 케이스 3D프린팅
- **더 깊은 소프트웨어**: 실제 DB(파이어베이스) · 알림 푸시 · 관리자 페이지
- **더 깊은 데이터·AI**: 쌓인 CSV로 예측해 보기 · 이미지 분류 정확도 올리기
- **동아리·대회로**: 교내 실사용 운영, 메이커 대회·창업 동아리 출품 — **우리 시스템은 이미 포트폴리오!**

## ✅ 7주의 기록 — 우리가 걸어온 길

- 1주: LED 하나 켜는 것부터 시작했어요
- 2~3주: 센서를 읽고, 웹을 만들고, 서버가 되었어요
- 4주: 클라우드에 데이터가 남기 시작했어요
- 5~6주: 설계하고, 연결하고, 통계까지 완성했어요
- 7주: 남 앞에서 증명했어요 — **완주!** 🎓

여러분은 이제 IoT 시스템을 **'전 구간'** 만들어 본 사람 — 웹 · 하드웨어 · 클라우드 · 데이터를 모두 통과했어요.

> 기술은 도구이고, 여러분이 배운 것은 '문제를 해결하는 방법'이에요 — 어디에든 다시 쓸 수 있어요.
