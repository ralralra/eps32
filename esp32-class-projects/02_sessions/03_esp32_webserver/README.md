# 3회차 — ESP32 웹서버: 폰으로 보드를 조종해요

> 지난주 만든 웹페이지를 ESP32에 심어요 — 인터넷 없이, 내 보드가 곧 서버예요.
> **ESP32 웹서버 · 4교시 (50분 수업 + 10분 휴식)**

## 오늘의 목표

- [ ] ESP32를 AP 모드로 설정해 자체 WiFi를 만들고, 스마트폰으로 접속할 수 있다
- [ ] 웹 버튼 → ESP32 → LED·LCD로 이어지는 '제어 흐름'을 구현할 수 있다
- [ ] 센서값 → ESP32 → 웹 화면으로 이어지는 '표시 흐름(폴링)'을 구현할 수 있다

## 시간표

| 교시 | 활동 |

## 1교시 — 서버의 정체, 그리고 AP 모드

### 서버란? 한 문장이면 충분해요

> **서버 = 요청을 받으면 응답하는 프로그램**

- 식당에 비유: 손님(스마트폰)이 주문(요청)하면 → 주방(서버)이 음식(응답)을 내줘요
- 웹에서는: 브라우저가 주소를 열면(요청) → 서버가 HTML을 돌려준다(응답)
- 유튜브·네이버도 똑같아요 — 규모만 클 뿐, '요청→응답' 구조는 완전히 동일
- **오늘의 선언: ESP32, 네가 오늘부터 서버다!** 우리 팀 전용 미니 유튜브 본사 🏢

### AP 모드 — ESP32가 공유기가 돼요

| 평소의 WiFi (Station 모드) | 오늘의 WiFi (**AP 모드**) |
|---|---|
| ESP32가 공유기에 '접속하는' 쪽 | ESP32가 WiFi를 '**만드는**' 쪽! |
| 공유기 → 인터넷으로 나가요 | 폰이 ESP32에 직접 접속 |
| 학교 WiFi가 막히면 곤란 | **인터넷이 전혀 필요 없어요** |
| 4회차에 이 모드를 써요 | 교실 어디서든 100% 동작 |

AP = Access Point(접속 지점). 폰 WiFi 목록에 **우리 팀 이름이 뜨는 순간**을 곧 보게 돼요.

### 따라하기 ① AP 모드 + 첫 웹서버

→ [`code/01_ap_hello/01_ap_hello.ino`](code/01_ap_hello/01_ap_hello.ino)

- `softAP()` 한 줄로 WiFi 생성 (⚠ 비밀번호 8자 이상!)
- `server.on("주소", 할일)`: 주소별 응답 등록
- `send(200, 형식, 내용)`: 200 = '정상 처리' 신호
- `handleClient()`: 손님 왔나 계속 확인
- **팀 번호로 WiFi 이름을 바꿔서** 사용 (Team2_IoT …)

### 따라하기 ② 스마트폰으로 접속하기

1. 코드를 업로드하고 잠시 기다려요 (WiFi 켜는 데 몇 초)
2. 폰 설정 → WiFi 목록에서 `Team1_IoT` 찾기
3. 비밀번호 `12345678` 입력 후 연결 — **'인터넷 연결 없음' 경고가 떠도 정상!** 연결 유지 선택
4. 브라우저 주소창에 **`192.168.4.1`** 입력 (AP 모드에서 ESP32 주소는 항상 이 번호)
5. 화면에 `Hello ESP32!` 가 뜨면 성공! 📱→📡 내 폰이 내 보드의 웹페이지를 받아왔어요

> 🔧 폰이 자꾸 WiFi를 끊는다면? '모바일 데이터'를 잠시 끄면 해결 — 폰이 인터넷 되는 쪽으로 도망가는 것.
> ✓ **옆 사람 폰으로도 접속시켜 보기** — 여러 명이 동시에 접속돼야 진짜 서버! 접속 화면 캡처 → 오늘의 첫 산출물.

## 2교시 — 웹 버튼으로 진짜 LED 켜기

### 엔드포인트 — 주소에 명령을 실어요

주소 뒤에 붙는 경로가 곧 '요청서의 제목'!

| 주소 | 의미 |
|---|---|
| `192.168.4.1/` | 대문 — 웹페이지(HTML)를 주세요 |
| `192.168.4.1/led/on` | 명령 — LED를 켜 주세요 |
| `192.168.4.1/led/off` | 명령 — LED를 꺼 주세요 |
| `192.168.4.1/sensor` | 질문 — 지금 센서값을 알려 주세요 (3교시) |

`server.on()`을 주소마다 하나씩 등록하면 돼요 — **이것이 모든 웹 API의 원리**이고, 다음 주 구글 Apps Script도 똑같아요.

### 따라하기 ③ 내 HTML 심기 + LED 엔드포인트

→ [`code/02_led_remote/02_led_remote.ino`](code/02_led_remote/02_led_remote.ino)

- `R"rawliteral( )"` : 따옴표 걱정 없이 HTML을 통째로 담는 그릇 — **지난주 HTML을 여기에 붙여넣기!**
- 버튼의 `onclick`이 `fetch('/led/on')` 으로 변경 — 지난주의 '빈칸'이 채워졌어요!
- 요청이 오면 `digitalWrite` 실행 → **진짜 LED가 켜져요**

### 실습 지도 — 코드는 4개 구역, 구역만 지키면 절대 안 꼬여요

1. **구역 1 · 맨 위**: `#include`와 전역 선언 — WiFi.h, WebServer.h, 핀 `#define`, `PAGE[]`(HTML 그릇)
2. **구역 2 · setup() 안**: 준비와 등록 — `pinMode` → `softAP` → `server.on` 등록들 → `server.begin()` 이 순서로
3. **구역 3 · loop() 안**: 단 한 줄 — `server.handleClient();` (⚠ delay를 넣으면 반응이 느려져요!)
4. **구역 4 · HTML 수정은 PAGE 안에서만** — 고치면 반드시 재업로드

### 방금 일어난 일 — 전체 흐름 복기 ✍

폰 화면 터치부터 LED 점등까지 0.1초 동안의 여행:

```
폰에서 [LED 켜기] 터치  (onclick 발동)
      ▼
JS가 fetch('/led/on') 전송  (WiFi 전파를 타고)
      ▼
ESP32가 요청 접수  (server.on("/led/on") 매칭)
      ▼
digitalWrite 실행 → LED 점등!  (그리고 "OK" 응답 회신)
```

> 📝 **이 그림을 종이에 직접 그려보는 것이 오늘의 평가 과제.** 모든 IoT 제품(스마트 전구, 로봇청소기 앱)이 이 구조예요.

## 3교시 — 센서값을 웹 화면에: 폴링

이번엔 반대 방향: 보드의 감각을 폰 화면으로!

### 따라하기 ④ /sensor 엔드포인트 — JSON으로 응답

JSON = 데이터를 주고받는 표준 문법 `{ "이름": 값 }` → [`code/03_sensor_json/03_sensor_json.ino`](code/03_sensor_json/03_sensor_json.ino)

- JSON은 '이름표 붙은 값 묶음' — `temp`라는 이름표에 25.3이 담겨 와요
- 같은 방식으로 조도·소음·거리도 추가 — 키트의 어떤 센서든 JSON에 담을 수 있어요
- **폰 브라우저에서 `192.168.4.1/sensor` 직접 열어 확인!** `{"temp":25.3,"humi":48.0}` 모양 그대로여야 해요
- 센서에 입김을 불고 새로고침 → 습도 숫자가 확 오르면 센서도, 서버도 정상!

> ✓ 구간 점검의 습관: **화면(JS)을 의심하기 전에 재료(JSON)부터 확인** — 디버깅 시간이 절반으로 줄어요.

### 따라하기 ⑤ 웹에서 2초마다 자동 갱신 — 이것이 '폴링(polling)'

```html
<p>온도: <span id="t">--</span> ℃</p>
<script>
async function update() {
  const res = await fetch("/sensor");
  const d = await res.json();
  document.getElementById("t").innerText = d.temp;
}
setInterval(update, 2000); // 2초마다 반복 실행
</script>
```

- 폴링 = 주기적으로 '지금 어때?' 물어보기
- 실시간처럼 보이지만 사실은 **2초짜리 질문의 반복**
- 다음 주 클라우드 구조도 같은 폴링 원리예요

### 🎯 미션 ① 미니 기상 스테이션

**목표**: 폰 화면에 교실의 온도·습도가 2초마다 자동으로 갱신되어 표시

완성 조건:
- [ ] `/sensor` 엔드포인트가 JSON 정상 응답
- [ ] 화면 숫자가 자동으로 변함 (센서에 입김 불어 테스트!)
- [ ] 심화: 조도 값도 추가하고, 온도 28℃ 이상이면 화면 배경을 빨갛게

> 💡 힌트: `if (d.temp > 28) document.body.style.background = "#FBECEC";`

## 4교시 — 종합 미션: 내 손안의 리모컨

### 🎯 종합 미션 ② 내 손안의 리모컨

**목표**: 폰 한 화면에서 — LED 켜기/끄기 + 입력한 글자를 교실 LCD로 전송 + 온습도 실시간 표시
→ 완성 예시: [`code/04_full_remote/04_full_remote.ino`](code/04_full_remote/04_full_remote.ino)

완성 조건:
- [ ] LED 켜기/끄기 버튼
- [ ] 글자 입력창 + [보내기] 버튼 → 그 글자가 교실 LCD에 표시 (`/lcd` 엔드포인트)
- [ ] 온습도 2초 자동 갱신
- [ ] 팀원 폰 최소 2대에서 동시 접속 성공 — **서로 LCD에 인사를 보내 봐요!**

> 💡 힌트: **하나 완성 → 테스트 → 다음 추가!** LCD는 `server.arg("msg")`로 글자를 꺼내 `lcd.print`, 웹은 `fetch('/lcd?msg=' + 입력값)` — 한 번에 다 넣으면 어디서 틀렸는지 못 찾아요.

### 한계 토론 — 그런데 말입니다 (조별 토론 5분)

| 지금 시스템의 한계 | 무엇이 있으면 해결될까? |
|---|---|
| 전원을 끄면 측정 기록이 모두 증발 | 항상 켜져 있는 **저장소**가 필요 |
| 통계? 그래프? 저장이 안 되니 불가능 | ↑ 같은 답 |
| ESP32 WiFi 범위(교실) 밖에선 접속 불가 | 어디서든 접속되는 '**중간 장소**'가 필요 |

**= 클라우드!** 다음 주: 구글 시트가 그 역할을 맡아요. 기술은 언제나 '불편함'에서 태어나요.

### 심화(선택) — LittleFS로 파일 분리

빨리 끝난 팀만! HTML을 코드 문자열에 넣으니 수정할 때마다 재업로드 — LittleFS(ESP32 내부의 작은 저장공간)에 HTML을 '파일 그대로' 넣으면 화면(HTML)과 두뇌(C 코드)를 분리할 수 있어요. 업로드 도구 설치가 필요하므로 교사에게 요청.

## 🔧 자주 막히는 문제 해결소 — 3회차

| 증상 | 해결 |
|---|---|
| 폰 WiFi 목록에 팀 WiFi가 안 보여요 | 업로드 성공 확인 → 보드 리셋(EN 버튼) → **비밀번호 8자 이상인지** 확인 (짧으면 AP 생성 실패) |
| 연결은 됐는데 페이지가 안 열려요 | 주소 `192.168.4.1` 정확히 입력 (http:// 없이 OK), 폰 **모바일 데이터 끄기** |
| 버튼은 눌리는데 LED가 안 켜져요 | 폰 브라우저에서 `192.168.4.1/led/on` 을 **주소로 직접 열어보기** — 열리면 JS 문제, 안 열리면 server.on 등록 문제 |
| rawliteral 부분에서 컴파일 에러 | `R"rawliteral(` 과 `)rawliteral"` 짝 확인 — HTML 안에 `)rawliteral"` 글자가 있으면 안 됨 |
