## 1교시 — 무엇을, 어디까지 만들 것인가

'다 만들고 싶다'는 마음이 프로젝트를 망치는 첫 번째 이유예요.

### 왜 설계를 먼저 할까

1. **만들면서 정하면**: 웹 담당과 보드 담당이 서로 다른 것을 만들고 있어요 → 합칠 때 대참사
2. **화면과 데이터를 '먼저 합의'하면**: 각자 따로 만들어도 나중에 딱 맞물려요 — 협업의 기술
3. **범위를 '먼저 제한'하면**: 필수 기능부터 완성 → 시간이 남으면 추가. 반대로 하면 아무것도 완성 못 해요
4. 실제 개발 회사도 똑같아요: 기획서 → 디자인 시안 → DB 설계 → 개발 — 오늘 그 과정을 압축 체험!

### 5개 프로젝트 — 필수 기능과 우회 전략 요약

> 우회는 미완성이 아니라 **'설계 전략'** — 실제 개발의 프로토타이핑 기법이에요.

| 프로젝트 | 필수 기능 | 우회 전략 | 핵심 부품(고릴라셀) |
|---|---|---|---|
| ① 매점 | 상품 판매·재고 차감·인기 순위 | RFID → **웹 버튼 + 무게로 실물 감지** | LCD·무게·진동 |
| ② 키오스크 | 메뉴·장바구니·주문번호·매출 | 실결제 → **가상 포인트** | LCD(주문번호)·소리 |
| ③ 우산 대여 | QR 대여·반납·이용 통계 | GPS → **초음파·무게 거치 감지** | 초음파·무게·LCD |
| ④ 화분 | 수분·빛·온습도 측정·급수 관리 | 펌프 → **급수 알림(LCD·웹)** | 토양습도·빛·DHT·물온도 |
| ⑤ 출석 | 체크인·현재 인원·출석부 | NFC → **웹 QR + 홀 도어 감지** | 홀(도어)·LCD·심박(체험) |

**모든 팀의 공통 뼈대 = 4회차 템플릿 (웹앱 ↔ Apps Script ↔ 시트 ↔ ESP32). 다른 건 '내용물'뿐!**

### 따라하기 ① 기능 명세서 작성

워크시트([`worksheets/기능명세서.md`](worksheets/기능명세서.md))의 표를 채워요 — 문장은 반드시 **'검증 가능하게'**:

1. **필수 기능** (이것 없으면 실패) — 예: '웹에서 상품을 누르면 재고가 1 줄고 시트에 기록된다' ← 되고/안 되고를 확인할 수 있는 문장!
2. **추가 기능** (시간 남으면) — 예: '재고 3개 이하면 웹 화면·LED로 경고'
3. **제외 기능** (이번엔 안 만들어요) — 예: '실제 결제'. **제외를 적어두는 것도 훌륭한 설계 결정!**
4. **기능마다 사용 부품 명시** — 예: 주문 알림 → LCD(번호 표시) + LED(깜빡임). 부품이 안 적힌 기능은 아직 설계가 아니에요

> ✓ 검토 기준: **다른 팀이 명세서만 읽고도 무엇을 만드는지 정확히 알 수 있나?**

### 역할 분담 — 4개의 모자

모두가 모든 걸 조금씩 하되, '책임자'는 정해요. (책임자 = 그 부분이 막혔을 때 **가장 먼저 달려드는 사람**. 혼자 하라는 뜻이 아니에요!)

| 모자 | 하는 일 |
|---|---|
| 🌐 웹 담당 | AI Studio 프롬프트 · 화면 다듬기 · fetch 연결 |
| 🔧 하드웨어 담당 | 배선 · 센서 테스트 · ESP32 코드 · 핀맵 관리 |
| 📊 데이터 담당 | 시트 구조 · Apps Script · 통계 · CSV |
| 🎤 발표·기록 담당 | 진행 기록 · 사진/영상 촬영 · 7회차 발표 준비 |

## 2교시 — 화면 설계와 데이터 설계

**종이 스케치 한 장이 코드 백 줄을 아껴요.**

### 따라하기 ② 와이어프레임 — 종이에 화면 그리기

1. 폰 화면 비율의 네모를 그려요 (우리 앱은 모바일 우선 — 세로로 길게)
2. 화면에 들어갈 요소를 네모·글자로 배치 — 제목 / 큰 숫자(재고·인원) / 버튼(크게!) / 목록 / 통계 영역
3. 화면이 여러 개면 화살표로 이동 연결 — 예: 메뉴 화면 → [주문] → 완료 화면(주문번호)
4. **버튼마다 '누르면 무슨 일이?' 메모** — 이 메모가 그대로 Apps Script의 할 일 목록이 돼요

> 예쁘게 그릴 필요 전혀 없음 — **팀원 모두가 같은 그림을 머리에 갖게 하는 것**이 목적.
> 완성되면 밝은 곳에서 정면 촬영 → 팀 드라이브 '설계' 폴더에 `팀○_화면스케치_v1` 업로드.
> ✓ 검증: 옆 팀에게 사진만 보여주고 **설명 없이 이해되는지** 테스트 — 통과하면 프롬프트로 옮길 준비 완료!

### 따라하기 ③ 시트 설계 — 우리 팀 데이터베이스

**표의 열 제목을 정하는 것 = 데이터베이스 설계** → 워크시트: [`worksheets/시트설계서.md`](worksheets/시트설계서.md)

| 프로젝트 | 시트(탭) 구성 | 열 구성 예시 |
|---|---|---|
| ① 매점 | 상품 / 판매로그 | 상품명·가격·재고 / 시각·상품명·수량 |
| ② 키오스크 | 메뉴 / 주문로그 | 메뉴명·가격 / 시각·주문번호·내역·합계 |
| ③ 우산 | 우산현황 / 대여로그 | 우산번호·상태 / 번호·대여시각·반납시각 |
| ④ 화분 | 환경로그 / 급수로그 / 설정 | 시각·수분·온도·습도·조도 / 시각·자동여부 / 임계값·명령 |
| ⑤ 출석 | 학생명부 / 출입로그 | 이름·QR값 / 이름·입실시각·퇴실시각 |

설계 원칙:
1. **기록(로그)은 지우지 않고 계속 쌓아요**
2. **현재 상태와 기록을 다른 탭으로 분리해요**

## 3교시 — AI Studio로 웹앱 제작

스케치를 '말'로 번역하면 AI가 '화면'으로 번역해요.

### 좋은 프롬프트 작성법 — 4요소

1. **무엇을 만드는지 한 문장**: "학교 매점 관리 웹앱을 만들어줘"
2. **화면 요소를 스케치 순서대로**: "상단에 제목, 상품 카드 6개(이름·가격·남은 재고·판매 버튼), 하단에 오늘 판매 수·인기 순위"
3. **공통 디자인 규칙 (전 팀 동일)**: "파란 계열, 모바일 우선, 버튼 크게, 아이콘 사용, 큰 글씨"
4. **동작 규칙**: "판매 버튼을 누르면 재고가 1 줄고 확인 메시지 표시"

> ❌ 나쁜 프롬프트: "매점 앱 이쁘게" — 모호하게 시키면 모호한 결과. **스케치의 모든 요소를 문장에 담자.**
> 팀별 프롬프트 예시 원문: [`worksheets/프롬프트_예시.md`](worksheets/프롬프트_예시.md)
> AI Studio(New App) 사용법 + 폰 설치(PWA) 전체 가이드: [`01_docs/ai_studio_webapp_guide.md`](../../01_docs/ai_studio_webapp_guide.md)

### 진행 순서

1. 완성한 프롬프트를 **문서에 먼저 저장** — 프롬프트도 팀의 산출물!
2. AI Studio Build에 붙여넣고 생성
3. 스케치와 결과를 나란히 놓고 비교 — 빠진 요소·다른 배치·이상한 문구를 목록으로
4. 차이 목록을 **하나씩** 추가 지시로 해소 (뭉뚱그리면 엉뚱하게 바뀌어요)
5. 1차 완성 화면을 캡처 — 발표의 'Before → After' 재료!

### 생성 후 해야 할 일 — AI는 초안, 검토는 사람

1. 스케치와 비교하며 다른 점 찾기 → 하나씩 추가 지시
2. **코드 보기로 구조 확인** — 2회차의 HTML·CSS·JS가 그대로! id 이름들을 기록해 두면 연결할 때 편해요
3. 화면 전환·버튼을 직접 눌러보기 — 동작 안 하는 버튼은 '아직 껍데기', 4교시에 연결
4. **과한 기능은 오히려 삭제 요청** — "로그인 기능은 빼줘". 범위 지키기는 여기서도 계속!

## 4교시 — 뼈대 연결과 가짜 데이터 테스트

진짜 센서 없이도 화면은 완성할 수 있어요.

### 따라하기 ④ Apps Script 뼈대 + 가짜 데이터(mock)

하드웨어보다 화면을 먼저 완성하는 개발 요령:

```javascript
// [Apps Script] 프로젝트용으로 확장 — action 파라미터로 '요청 종류' 구분
function doGet(e) {
  const a = e.parameter.action;
  if (a == "sell")   return sell(e);
  if (a == "stats")  return stats();
  if (a == "latest") return latest();
  return ContentService.createTextOutput("unknown");
}

// [웹앱 JS] 아직 시트가 비었다면 가짜 데이터로 화면부터 확인:
const mock = {"cola": 7, "chip": 3};
renderStock(mock); // 화면 그리기 테스트
```

- `action` 분기는 3회차 엔드포인트의 응용 — `sell`·`stats` 같은 함수를 하나씩 채워가요 (6회차)
- 가짜 데이터로 화면 로직을 미리 검증 → 나중에 진짜 데이터로 교체만
- **오늘의 완성선: 화면이 그럴듯하게 '동작하는 척'까지!**

### 🎯 오늘의 산출물 체크 — 4종 세트

**목표**: 설계 문서 3종 + 웹앱 1차 버전. 다음 주는 이 설계도대로 '시공'만!

- [ ] AI Studio 웹앱 1차 버전 (가짜 데이터로 동작)
- [ ] 팀원 전원 폰에 PWA 설치 (4회차 방법)
